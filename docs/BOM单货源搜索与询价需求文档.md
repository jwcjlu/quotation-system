# BOM 单货源搜索与询价需求文档

基于多平台 BOM 配单、货源搜索与价格比对的功能需求。

---

## 1. 系统概述

本系统为电子元器件采购的 **BOM 单货源搜索与询价** 功能模块，支持用户导入 BOM 文件，自动解析物料清单；**每次配单会话可由用户勾选参与搜索/比价的货源平台**（见 §1.3、§2.3），在各所选平台上搜索或复用当日缓存，并完成配单与价格比对。

**核心流程（摘要）：** 上传 BOM → 解析 → **选定本次参与的平台** → **按型号查库** → 若对**每个所选平台**该型号均已「有报价或已确认无该型号」（当日）则就绪并进入配单，否则 **生成搜索任务并广播** → 任务模块派发 → **Agent 经任务心跳领取并执行** → 结果入库 → **通知配单**。

### 1.1 术语与两层逻辑（搜索缓存 vs 配单匹配）

为避免「仅型号查缓存」与「型号+封装+厂牌配单」混读，约定如下：

| 概念 | 作用范围 | 说明 |
|------|----------|------|
| **搜索缓存键** | 第 8 节后台 | **型号 + 货源平台 + 业务日**。用于判断是否已对该型号在该平台当日执行过搜索、是否可复用缓存；**不**把厂牌、封装作为缓存主键。 |
| **平台搜索结论** | 第 8 节 / §1.3 | 除报价列表外，须能表达 **「无该型号/无匹配」** 的终态（与「有报价」并列），二者之一即可使该平台对该型号**就绪**。 |
| **配单匹配** | 第 2.3、2.4 节前台 | 在已获得的各平台报价**候选集**中，按 **型号、封装、厂牌** 与价格策略筛选推荐行；若某平台为「无该型号数据」则该平台不参与该行推荐。 |
| **关系** | — | 先按 §1.3 对**所选平台**逐项达成就绪；再在「有数据」的平台中取候选集做配单匹配与比价。缺数据且未判「无匹配」时走第 8 节任务与 Agent。 |

### 1.2 主流程（九步）

以下为主干需求顺序，与 §6、§8 及 [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md) 一致。

| 步骤 | 环节 | 说明 |
|------|------|------|
| 1 | 用户上传 BOM 单 | 支持 Excel（.xlsx / .xls），入口见 §2.1 |
| 2 | 系统解析 BOM | 解析出**型号、厂牌、封装、订单量（数量）**等，规则见 §2.2；解析结果可编辑 |
| 3 | 按型号查库与就绪判定 | 以**型号**为键，针对**本次会话勾选的平台集合**查库；对每个所选平台，须已具备当日有效的 **「有报价」或「无该型号/无匹配」** 结论（见 §1.3）。**全部所选平台均满足时**可**直接进入配单**（§2.4）；否则进入步骤 4 |
| 4 | 生成搜索任务并广播 | 对**今日尚未就绪**的「型号 × 平台」生成搜索任务并持久化；发布**消息广播**（或等价事件），唤醒/通知搜索任务模块 |
| 5 | 搜索任务模块触发 | **定时轮询**与/或 **订阅步骤 4 的广播**；发现有可派发任务即进入派发 |
| 6 | 任务派发规则 | 将任务分配给**当前空闲且具备对应平台能力**的 Agent；任务通过 **Agent 任务心跳**（长轮询响应捎带，见架构 §4.2、§4.5）下发，与「脚本安装心跳」分离 |
| 7 | Agent 执行与上报 | Agent 接收任务载荷（`script_id`、`version`、参数等），执行采集脚本，通过**任务结果上报接口**回传；执行语义、超时、串行规则见 [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md) §4.4–§4.6 |
| 8 | 服务端入库并通知配单 | 服务端校验并标准化结果（§3.4）后**入库**；向配单域推送**数据就绪通知**（或事件），以便刷新报价并驱动/继续 **自动配单**（§2.4） |
| 9 | Agent 总体设计 | **须符合** [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md) 与 [分布式采集Agent-API协议](./分布式采集Agent-API协议.md)；双心跳、任务发现、结果上报、能力/队列匹配等不另起一套协议 |

### 1.3 配单会话平台选择与「型号就绪」条件

**平台选择（每次配单）**

- 用户在发起/进行配单时，**勾选本次需要搜索与比价的货源平台**（可多选）；仅对勾选平台查库、建任务、展示比价列。
- 当前可选平台标识（与实现/脚本约定一致，可扩展）：

| 平台 ID | 说明 |
|---------|------|
| `find_chips` | FindChips 等对应采集能力 |
| `hqchip` | HQChip 等对应采集能力 |
| `icgoo` | ICGOO |
| `ickey` | 云汉芯城（ickey / search.ickey.cn） |
| `szlcsc` | 立创商城（szlcsc） |

**型号就绪（允许进入配单）**

对 BOM 中**每一个型号**，在**本次勾选的全部平台**上，须同时满足：在该平台、该业务日下，已存在下列**之一**（均为「当日已搜索闭环」）：

| 结论类型 | 含义 |
|----------|------|
| **有数据** | 有可供比价的报价数据（一条或多条，结构化见 §3.4） |
| **无该型号数据** | 已完成对该型号的搜索，且平台侧**无匹配物料/无结果**（空结果），须以结构化「无匹配」结论入库，避免无限重搜 |

当**上述条件对「该型号 × 每一个所选平台」均成立**时，该型号视为**就绪**，可参与自动配单与展示；未就绪的平台组合继续走步骤 4–8。**未勾选的平台**不参与判定与展示。

**说明：** 「有数据」与「无该型号数据」互斥；执行失败、超时等**非终态**不视为就绪，规则见 §4 第 8 条、§8.6。

## 2. 功能模块

### 2.1 BOM 单导入功能

**功能描述：**
- 提供 BOM 文件导入入口，支持用户上传 BOM 单
- 支持 Excel 格式（.xlsx、.xls）

**核心功能点：**

| 功能项 | 说明 |
|--------|------|
| Excel 拖拽上传 | 支持将 Excel 文件拖拽至指定区域上传 |
| Excel 文件上传 | 点击「Excel文件上传」按钮选择本地文件 |
| 解析模式选择 | 支持选择预设解析模式或自定义列映射，见 2.2 |
| 模板下载 | 提供标准 BOM 模板，提升解析与配单准确率 |
| 历史 BOM 查看 | 查看历史配单记录 |

**界面结构：**
- 中央上传区：Excel 拖拽区域 + 上传按钮
- 底部：模板下载链接、历史 BOM 链接

---

### 2.2 BOM 单解析

**功能描述：**
- 用户导入 BOM 单后，系统按所选解析模式解析文件内容
- 从 BOM 中提取型号、厂牌、封装、数量、参数等字段

**解析模式选择：**

用户在上传 Excel 前或上传后，可选择解析模式，以适配不同来源的 BOM 格式：

| 解析模式 | 说明 | 适用场景 |
|----------|------|----------|
| 立创标准 | 按立创商城 BOM 模板列顺序解析 | 从立创导出的 BOM |
| 云汉标准 | 按云汉芯城 BOM 模板列顺序解析 | 从云汉导出的 BOM |
| 通用模式 | 自动识别表头，按列名匹配（型号、厂牌、封装等） | 常见 BOM 格式 |
| 自定义映射 | 用户手动指定：型号→A列、厂牌→B列、封装→C列… | 非标准格式 |

**自定义映射界面：**
- 展示 Excel 首行预览，供用户确认表头
- 下拉选择各字段对应的列（A/B/C… 或列名）
- 支持保存为自定义模板，下次复用

**解析规则：**

| 解析项 | 说明 |
|--------|------|
| 型号 | 从 BOM 中识别元器件型号（如 MP1658GTF-Z） |
| 厂牌 | 识别制造商/品牌（如 MPS、TI） |
| 封装 | 识别封装类型（如 TSSOP-16、LGA-16） |
| 数量 | 识别需求量 |
| 参数 | 其他技术参数（电压、电流等） |

**输出：**
- 解析后的结构化物料列表，供后续多平台搜索使用
- 支持用户对解析结果进行编辑、修正

---

### 2.3 多平台货源搜索

**功能描述：**
- **每次配单会话**由用户**勾选本次参与搜索与比价的平台**（§1.3），系统仅对所选平台查库、建任务、展示列与比价。
- 根据解析后的 BOM，在所选平台上获取报价或「无该型号」结论（§1.3），支持多平台并行或串行，汇总数据。

**可选平台（与 §1.3 一致）：** `find_chips`、`hqchip`、`icgoo`、`ickey`、`szlcsc`（展示名与 URL 由产品/实现配置，如立创、云汉、ICGOO、FindChips、HQChip）。

**界面要求：**
- 提供**平台多选**（复选框或等价控件），默认可全选或记忆上次选择；变更选择后，就绪判定与任务范围仅针对**当前勾选集合**。

**搜索流程（业务视角）：**
1. 遍历解析后的 BOM 物料列表
2. 对每个物料型号、**每个本次勾选的平台**，确认是否已有**当日**终态：**有报价**或**无该型号/无匹配**（见 §1.3、§8.1）；未满足则由后台生成任务并由 Agent 采集（§8），再汇总
3. 采集各平台返回的：型号、厂牌、封装、库存、货期、价格梯度、单价等；若无匹配则写入「无匹配」结论
4. 汇总各平台报价数据，供自动配单使用

**匹配与选型规则：**

各平台搜索后通常返回多条记录，系统需按以下规则筛选并选择一条作为推荐：

| 规则 | 说明 |
|------|------|
| 型号匹配 | 平台返回的型号与 BOM 解析出的型号一致 |
| 封装匹配 | 平台返回的封装与 BOM 解析出的封装一致 |
| 厂牌匹配 | 平台返回的厂牌与 BOM 解析出的厂牌一致 |
| 价格最低 | 在满足上述三项匹配的记录中，选择单价最低的一条 |

若某平台对该型号为「无该型号数据」（§1.3），则该平台不参与该物料的推荐；若有报价但无完全匹配记录，则该平台不参与该物料的推荐；若多平台均有可参与匹配的记录，则跨平台比较后取价格最低者。

**数据采集字段：**

| 字段 | 说明 |
|------|------|
| 型号 | 匹配到的型号 |
| 厂牌 | 供应商/制造商 |
| 封装 | 封装类型 |
| 库存 | 可用库存 |
| 货期 | 交货周期 |
| 起订量 | 最小起订量 |
| 价格梯度 | 阶梯价格（如 5+ ￥0.88 \| 50+ ￥0.74） |
| 中国香港交货 | 香港交货价格 |
| 内地交货(含增值税) | 内地含税价格 |

### 2.4 自动配单

**功能描述：**
- 根据多平台搜索报价数据，自动完成配单
- 各平台返回多条记录时，按 **型号、封装、厂牌均匹配** 且 **价格最低** 的规则选型（详见 2.3 匹配与选型规则）
- 支持按价格、库存、货期等策略选择最优货源
- 支持多平台货源混选（同一 BOM 中不同物料可从不同平台采购）

**配单策略（可配置）：**

| 策略 | 说明 |
|------|------|
| 价格优先 | 选择单价最低的货源 |
| 库存优先 | 选择库存充足的货源 |
| 货期优先 | 选择货期最短的货源 |
| 综合排序 | 综合价格、库存、货期等维度 |

**输出：**
- 每行物料对应推荐最优型号及来源平台
- 库存、货期、单价、小计
- 支持用户手动调整或确认

---

### 2.5 BOM 配单结果与价格比对

**功能描述：**
- 展示 BOM 匹配结果，支持按匹配状态筛选
- 每行默认显示系统推荐的最优匹配；支持「显示更多」展开，查看该物料在各平台搜索到的全部报价
- 展示每行物料的库存、货期、单价、小计

**核心功能点：**

#### 2.5.1 匹配状态筛选

| 状态 | 说明 |
|------|------|
| 全部 | 显示所有匹配结果（示例：26 条） |
| 完全匹配 | 系统自动匹配成功的型号（示例：20 条） |
| 待确认 | 需人工确认的推荐型号（示例：4 条） |
| 无法匹配 | 匹配失败的型号（示例：2 条） |

#### 2.5.2 表格列定义

| 列名 | 类型 | 说明 |
|------|------|------|
| 序号 | 数字 | 行号 |
| 原始需求 | 文本 | 用户上传的原始数据（如：MP1658GTF-Z MPS, 4804...） |
| 整理后需求 | 结构化 | 解析后的需求：型号、厂牌、封装、参数；支持编辑、删除 |
| 总需求量 | 数字 | 用户需求数量 |
| 结果 | 状态图标 | 红色 X：无法匹配；橙色 !：待确认；绿色 ✓：完全匹配 |
| 推荐最优型号 | 结构化 | 默认显示系统推荐的最优匹配；支持「显示更多」展开，展示该物料在各平台搜索到的全部报价结果 |
| 库存/货期/批次 | 结构化 | 库存数量、货期（如 5-7工作日）、批次 |
| 购买数量 | 输入框 | 支持步进；显示起订量、增量、整包数 |
| 单价(含税) | 金额 | 含税单价（如 ¥0.92） |
| 小计 | 金额 | 该行总价（自动计算） |
| 操作 | 按钮 | 加入购物车、更换型号、确认型号、**查看详情** |

#### 2.5.3 查看详情

**功能描述：**
- 每行被匹配的记录项增加「查看详情」按钮
- 点击后弹出/展开详情面板，展示该物料在**各平台搜索到的全部报价记录**
- 详情内容包含：平台名称、型号、厂牌、封装、库存、货期、价格梯度、单价等，便于用户对比各平台报价并手动更换选择

| 展示项 | 说明 |
|--------|------|
| 平台 | 立创商城、云汉芯城、ICGOO 等 |
| 各平台记录列表 | 该平台搜索到的所有匹配记录（含未选中的） |
| 当前选中 | 高亮或标记当前推荐的最优记录 |

#### 2.5.4 其他功能

| 功能 | 说明 |
|------|------|
| 查看重复物料 | 复选框，控制是否显示重复物料 |
| 显示配置 | 下拉菜单，控制表格列显示 |
| 价格波动提示 | 黄色提示：价格/库存可能波动，以结算为准 |
| 显示更多 | 每行「推荐最优型号」支持展开，点击后展示该物料在各平台搜索到的全部报价（型号、平台、库存、货期、单价等），便于对比与更换 |

## 3. 数据模型

### 3.1 原始需求（BOM 导入前）

```
原始需求: 用户 BOM 文件中的原始数据（如 "MP1658GTF-Z MPS, 4804..."）
```

### 3.2 解析模式枚举

| 值 | 说明 |
|----|------|
| szlcsc | 立创标准模板 |
| ickey | 云汉标准模板 |
| auto | 通用模式，自动识别表头 |
| custom | 自定义列映射 |

### 3.3 解析后需求（BOM 解析输出）

| 字段 | 类型 | 说明 |
|------|------|------|
| 型号 | 文本 | 元器件型号 |
| 厂牌 | 文本 | 制造商 |
| 封装 | 文本 | 封装类型 |
| 数量 | 数字 | 需求量 |
| 参数 | 文本 | 其他技术参数 |

### 3.4 平台报价数据（多平台搜索输出）

| 字段 | 类型 | 说明 |
|------|------|------|
| 来源平台 | 文本 | 平台 ID：`find_chips` / `hqchip` / `icgoo` / `ickey` / `szlcsc` 等（与 §1.3 一致） |
| 搜索结论 | 枚举 | **有报价**（`has_quotes`）或 **无该型号/无匹配**（`no_mpn_match`），与 §1.3 就绪判定一致；`no_mpn_match` 时可无报价行 |
| 推荐型号 | 文本 | 匹配到的型号 |
| 厂牌 | 文本 | 制造商 |
| 描述 | 文本 | 电压、电流、封装等 |
| 库存 | 数字 | 可用库存 |
| 货期 | 文本 | 如 5-7工作日 |
| 批次 | 文本 | 批次信息 |
| 起订量 | 数字 | 最小起订量 |
| 增量 | 数字 | 增量单位 |
| 整包数 | 数字 | 整包数量 |
| 价格梯度 | 文本 | 如 5+ ￥0.88 \| 50+ ￥0.74 |
| 中国香港交货 | 文本 | 香港交货价格 |
| 内地交货(含增值税) | 文本 | 内地含税价格 |
| 单价(含税) | 金额 | 含税单价 |
| 小计 | 金额 | 购买数量 × 单价 |

**说明：** `搜索结论=no_mpn_match` 时，可无后续报价字段或仅记录占位；`has_quotes` 时须满足业务必填项校验。

### 3.5 配单结果（自动配单输出）

| 字段 | 类型 | 说明 |
|------|------|------|
| 推荐型号 | 文本 | 自动选中的最优型号 |
| 厂牌 | 文本 | 制造商 |
| 来源平台 | 文本 | 选中的货源平台 |
| 描述 | 文本 | 型号描述 |
| 库存/货期/批次 | 结构化 | 同 3.3 |
| 购买数量 | 数字 | 用户确认的采购数量 |
| 单价(含税) | 金额 | 含税单价 |
| 小计 | 金额 | 该行总价 |

### 3.6 匹配状态枚举

| 值 | 含义 |
|----|------|
| 完全匹配 | 自动匹配成功 |
| 待确认 | 需人工确认推荐 |
| 无法匹配 | 匹配失败 |

### 3.7 搜索任务与当日缓存（与第 8 节对应）

库表字段以实现/DB 设计为准；需求层建议至少区分：

| 实体 | 说明 |
|------|------|
| 搜索任务 | **型号、目标平台/能力**、`biz_date`（业务日）、状态、重试次数/上限、错误摘要、幂等键（建议 `mpn+platform+biz_date`）、时间戳 |
| 当日报价缓存 | 型号、**平台**、`biz_date`、**搜索结论**（有报价 / 无匹配）、与 3.4 对齐的标准化字段；可保留原始响应 ID 或快照引用便于审计 |

## 4. 业务规则

1. **BOM 导入**：用户导入 BOM 后自动触发解析，解析失败时提示用户检查格式或下载模板
2. **查库与就绪（对应 1.2 步骤 3）**：对每个解析出的型号，仅针对**本次会话勾选的平台**（§1.3）查库；对每个所选平台，须已有**当日**终态：**有报价**或**无该型号/无匹配**（§1.3）。**全部所选平台均达终态**时该型号可进入配单；否则对未闭环的「型号 × 平台」生成搜索任务（见 1.2 步骤 4–8）
3. **价格与库存**：以结算页为准，配单页展示价格可能波动
4. **匹配逻辑**：优先完全匹配（型号、封装、厂牌均一致），在匹配记录中选价格最低者；其次推荐待确认；无法匹配时提示用户修改
5. **购买数量**：需满足起订量，按增量单位调整
6. **两层逻辑**：搜索侧按第 8 节「仅型号 + 平台 + 日」去重与查库；配单侧按第 2.3 节在报价候选集中做封装/厂牌/价格选型（见 1.1）
7. **任务幂等**：同一自然日内，同一（型号，平台）仅对应一次对外搜索意图；任务创建与入队应幂等，避免重复插入
8. **失败可感知**：某平台搜索失败或超时，该物料在该平台当日无有效缓存，配单界面应展示为部分无结果或「待重试/仅其他平台」，具体交互与第 2.5 节状态枚举对齐
9. **跨日数据**：非当日缓存不得用于自动配单；用户若需历史报价仅作参考，须单独产品定义（当前默认不用于自动推荐）

## 6. 业务流程总览

### 6.1 与 §1.2 九步主流程对齐（推荐）

```
  [1] 上传 BOM          [2] 解析（型号/厂牌/封装/订单量…）
            │                        │
            └──────────┬─────────────┘
                       ▼
            [3] 按型号查各平台当日缓存
                       │
         ┌─────────────┴─────────────┐
         │ 所选各平台均已当日终态     │ 否
         ▼                           ▼
   进入 [配单]              [4] 生成搜索任务 + 消息广播
   （§2.4）                           │
                                       ▼
                    [5] 任务模块：定时 / 订阅广播 → 待派发队列
                                       │
                                       ▼
                    [6] 派发给「空闲 + 有能力」的 Agent（任务心跳捎带）
                                       │
                                       ▼
                    [7] Agent 执行脚本 → 任务结果上报
                                       │
                                       ▼
                    [8] 服务端入库 → 通知配单 → 汇总后继续/刷新配单
```

**Agent 行为**统一遵循 §1.2 步骤 9 及 [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md)。

### 6.2 用户视角简图（与前台章节对应）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────────────┐    ┌─────────────┐
│ BOM 上传     │ -> │ BOM 解析     │ -> │ 报价就绪后配单        │ -> │ 结果/比价    │
└─────────────┘    └─────────────┘    └─────────────────────┘    └─────────────┘
```

后台在「报价就绪」前可能经历：查库 → 建任务 → 广播 → 心跳派发 → Agent 采集 → 入库 → 通知（§1.2）。

## 8. 货源数据查询、任务编排、广播与当日有效规则

本节展开 **§1.2 步骤 3–8**，与 2.3「多平台货源搜索」及前台配单对应；Agent 侧行为以 [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md) 为准。

### 8.1 解析 BOM 后：按型号查库与「所选平台当日终态」

| 项 | 说明 |
|----|------|
| 触发时机 | BOM 解析完成，且用户已选定**本次参与的平台集合**（§1.3）。**缓存与搜索任务去重键为「型号 × 平台 × 业务日」**（不按厂牌、封装作为查库/建任务主键，见 1.1）。 |
| 查库 | **按型号**查询**所选各平台**在库中的当日报价/结论；须能区分 **有报价** 与 **无该型号（无匹配）** 两种终态（§1.3、§3.4「搜索结论」）。 |
| 就绪判定（进入配单） | **当且仅当**对该型号，**每一个所选平台**均已具备**当日**终态（有报价 **或** 已记录无匹配），该型号可**不经过进一步采集**进入配单（§2.4）；否则视为未就绪。 |
| 未就绪 | 对仍缺终态的「型号 × 平台」**幂等生成搜索任务**（同一组合同日不重复建任务，见 8.5）；并发出 **§8.2 消息广播**。 |

### 8.2 搜索任务模块：广播、定时与派发

| 项 | 说明 |
|----|------|
| 消息广播 | 新任务入队后发布**广播/事件**（实现可选消息队列、Pub/Sub、内事件总线），使搜索任务模块不必仅依赖轮询即可感知新任务（对应 1.2 步骤 4–5）。 |
| 触发方式 | 模块以 **定时扫描** 与/或 **订阅广播** 两种方式之一或组合触发派发逻辑（可配置）。 |
| 派发条件 | 存在待派发任务；存在 **在线且空闲**、且 **具备该平台脚本能力**（`script_id`/标签/队列等与架构 §4.6 一致）的 Agent。 |
| 下发通道 | 任务随 **任务心跳**（长轮询响应）返回给 Agent，**不**与脚本安装心跳混用（架构 §4.2、§4.5）；与 [agent-server-实现说明](./agent-server-实现说明.md) 中已实现接口对齐。 |

### 8.3 Agent 回传、入库与通知配单

| 项 | 说明 |
|----|------|
| 回传 | Agent 通过 **任务结果上报接口** 提交结果（字段与 [分布式采集Agent-API协议](./分布式采集Agent-API协议.md) 一致）；载荷中含结构化 `result`（如各平台报价 JSON）。 |
| 入库 | 服务端校验并映射为 §3.4 字段，**写入数据库**，并写入**业务日**以支持 8.4。 |
| 通知配单 | 入库成功后向配单/会话层发送**数据就绪通知**（事件、WebSocket、轮询标记等，实现由《BOM配单系统架构设计》定），以便界面刷新、触发或继续 **自动配单**（1.2 步骤 8）。 |
| 用途 | 供当前 BOM 会话及其它会话按型号复用当日缓存，减少重复采集。 |

### 8.4 搜索数据仅当日可用

| 项 | 说明 |
|----|------|
| 「当天」定义 | **自然日 0:00–24:00**（以系统/服务器所用时区为准，需在部署配置中明确）。 |
| 规则 | 通过本流程搜索并入库的数据，**仅在该自然日内**可作为配单/展示的有效货源数据使用。 |
| 查询 | 8.1 中「从数据库查询」必须限定为**当日数据**（与入库时写入的日期字段一致）。 |
| 跨日 | 超过当日的历史数据不用于自动配单命中；若需继续使用，应触发**新的搜索任务**或按产品策略归档/标注为「仅供参考」（若后续版本需要，再单独定义）。 |

### 8.5 已确认规则摘要

| 规则 | 说明 |
|------|------|
| 当日边界 | **自然日 0:00–24:00**（时区与系统配置一致）。 |
| 缓存/任务主键 | **型号 × 货源平台 × 业务日**（不按厂牌、封装作为查库或建任务主键；配单匹配仍见 1.1）。 |
| 同平台去重 | **同一型号在同一货源平台上，每个自然日只执行一次对外搜索**；已有当日结果或已存在对应任务时，不重复建任务。 |
| 就绪与配单 | **本次勾选的全部平台**对该型号均有**当日终态**：**有报价**或**无该型号/无匹配**（见 §1.3、8.1）；否则先走采集闭环；执行失败/超时等非终态见第 4 条第 8 款、§8.6。 |

### 8.6 搜索任务状态与失败处理（建议）

| 状态 | 说明 |
|------|------|
| 待派发 | 任务已持久化，等待调度分配给具备对应平台能力的 Agent |
| 已派发 / 执行中 | 已交给 Agent；可与心跳、租约结合（参见架构与 `agent-server` 实现） |
| 成功 | 结果已校验并入库，当日可按型号命中缓存 |
| 失败 | 可记录错误原因；允许**有限次数自动重试**（次数与退避策略可配置），仍属同一「当日一次搜索意图」下的重试，不新增重复对外搜索配额（与 8.1 幂等一致） |
| 取消 | 用户撤销或系统丢弃（如 BOM 会话关闭，规则由产品定） |

**说明**：具体状态命名与是否与采集 Agent 全局任务表合并，由《BOM配单系统架构设计》与数据库设计定稿；本节约束业务语义。

### 8.7 与 Agent / 服务端接口的衔接

- **总体设计**：双心跳、脚本 ZIP、`script_id`+`version`、任务心跳捎带任务、结果上报、租约/离线判定、能力匹配等均以 [分布式采集Agent需求与架构](./分布式采集Agent需求与架构.md) 与 [分布式采集Agent-API协议](./分布式采集Agent-API协议.md) 为准；BOM 货源搜索为其中一类业务任务（如按平台划分不同 `script_id`）。
- **实现对照**：当前服务端行为见 [agent-server-实现说明](./agent-server-实现说明.md)（心跳路径、409 租约等）。
- **生产入队**：由业务在「所选平台尚未全部达当日终态」时写入任务/队列；**禁止**生产依赖仅开发用途的 `dev/enqueue`。

### 8.8 Agent 执行与并发（引用架构）

- 同一 Agent 上 **同一 `script_id` 串行**（架构 §4.5）；不同 `script_id` 在满足资源前提下可并行。
- 超时、进程组终止、日志与 `result` 大小限制等见架构 **§6.2、§6.6** 与 API 协议。

## 9. 方案对比与取舍（头脑风暴）

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 全实时 | 每次配单直连各平台 | 数据新 | 慢、易被限流、成本高、难运维 |
| B. 仅静态库 | 不抓实时价，只用历史或手工维护库 | 快、稳 | 难以满足「询价」与当日有效约束 |
| **C. 混合（本文档）** | 当日缓存优先；miss 则任务 + Agent 拉取并入库；自然日失效 | 平衡成本、合规与体验 | 需任务调度、幂等与跨日策略 |

**推荐**：采用 **方案 C**，并以第 8 节与 1.1 两层逻辑落地。
