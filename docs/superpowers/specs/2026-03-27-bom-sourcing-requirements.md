# BOM 单管理与多平台配单 — 需求要点（产品规格输入）

> **文档性质：** 产品侧目标能力说明，可与当前代码状态独立看待；实现前以本文件为 **WHAT**，以 [2026-03-27-bom-sourcing-design.md](./2026-03-27-bom-sourcing-design.md) 为 **HOW（策略与状态机等）**，以 [../plans/2026-03-27-bom-sourcing-implementation.md](../plans/2026-03-27-bom-sourcing-implementation.md) 为落地任务拆解。

---

## 项目语境（简要）

本仓库 **BOM 应用 API 与相关实现可能曾下线或未完成**；下列要点描述的是 **应交付的产品能力**。恢复或新建 BOM 域时，须将 API、数据模型与本需求及设计 spec 对齐。

---

## 1. BOM 单主体与客户信息

- **新建 BOM**：不仅含物料行，还需 **客户/商机维度**，至少包含以下字段（可与 `bom_session` 等表对齐实现）：
  - **公司/客户**：公司名称（或客户名）
  - **联系**：邮件、电话、**联系人**
  - **内部管理**：**备注**、**内部单号**（可与外部单号区分，便于检索）
- **Excel 导入**：支持从表格导入 **行级明细**，字段至少覆盖：型号、厂牌、封装、参数、数量等；列映射、必填校验与错误行反馈（行号 + 原因）见设计 spec §6。

---

## 2. BOM 生命周期（CRUD）

- 对 **整张 BOM 单** 支持增删改查，含 **列表、详情、状态展示**。
- **列表能力（V1）**：支持 **分页**、**筛选**（至少按状态、日期、客户/内部单号等可组合）、**排序**（至少按更新时间、创建时间；具体可排序字段在 API spec 中列明）。
- **删除策略**：采用 **软删**（逻辑删除，保留主键与审计所需字段）；删除后 **不再** 对新搜索/配单生效，但 **历史任务记录、报价缓存与历史报价数据保留**（与 §4 一致）。具体列名（如 `deleted_at`）在 DDL 与 API 中约定。

---

## 3. 搜索任务模型（MPN × 平台）

- **平台集合**：ickey、find_chip、hqchip、icgoo、szlcsc（可配置扩展）；存库 ID 与设计 spec §1 对齐（如 `find_chip` → `find_chips`）。
- **任务粒度**：**每个平台 × 每个型号（行）= 一条独立搜索任务**（笛卡尔积；同一行多平台并行多任务）。
- 任务需有独立状态（待执行、执行中、成功、失败、取消等），便于轮询与重试。  
  → **状态枚举与转移** 见设计 spec §3。

---

## 4. BOM 变更与任务同步

- 当 BOM **增行、删行、改行**（型号/数量/厂牌/封装/参数等影响搜索或配单的字段）或 **平台勾选变化** 时，需 **增量调整** 对应搜索任务：
  - **新增**：为新增行补建 `pending` 任务（与设计 spec §4 一致）。
  - **删除行 / 取消平台 / 改行导致旧任务失效**：对不再有效的搜索任务执行 **作废**（如状态 `cancelled`，软标记为主，**不做物理删除**）。
  - **缓存与历史报价**：**报价缓存（如 `bom_quote_cache`）继续保留**；**历史报价/历史配单结果保留**（用于审计与对比），新配单仅基于 **当前有效任务与当前平台勾选** 下的数据；改行后按新 MPN/平台 **新建任务并重新抓取** 时，旧缓存键可因 `mpn_norm`/业务日等自然区分，**不强制删除历史缓存行**。
- 若实现层与设计 spec §4 有细节差异，以 **本文「作废 + 缓存与历史保留」** 为产品验收准则。

---

## 5. 「数据已准备」就绪条件

- **就绪检查方式（V1）**：以 **定时任务** 为主，按固定间隔扫描会话下任务聚合状态（可辅以事件触发作为优化，但 **必须具备定时兜底**，避免漏推状态）。
- **条件**：当 **所有型号** 在 **所有已选平台** 上搜索均达到设计 spec 定义的 **平台终态** 时，将会话置为 **数据已准备**（或等价状态名）。
- **部分平台失败**、**超时**、**手动跳过** 是否仍算「已准备」——见设计 spec §2 决策表与 `readiness_mode`（默认宽松模式与需求不冲突）。

---

## 6. 配单（匹配与择优）

- **门禁**：会话须达到 **数据已准备**（及 `readiness_mode` 约定）后，方可发起配单相关接口。
- **部分行配单（V1）**：允许对 **BOM 行子集** 执行配单；仅处理所选行，其余行状态不变；**行级就绪** 判定须与设计 spec §7 扩展说明一致（实现时在 API spec 中写明请求体与错误码）。
- **匹配规则**：对 **型号、封装、参数** 等与报价侧的对应字段采用 **完全匹配**：在 **同一规范化规则** 下字符串相等（或结构化字段逐项相等）；**不支持** 模糊匹配、相似度或「优先级兜底」作为 V1 默认行为。**厂牌** 在规范化后须经 **别名表映射到规范名** 再比较是否一致（见 [2026-03-28-bom-match-currency-mfr-design.md](./2026-03-28-bom-match-currency-mfr-design.md) §2）。
- **硬约束**：**库存 ≥ 客户要求数量**（数量口径与报价源字段在实现/API spec 中固定）。
- **优化目标**：在满足约束的候选中选取 **价格最低** 的项；**可比单价相同**（含舍入后相等）时 **交期更短者优先**，次键见该文档 §1.10。**价格口径、币种、排序键与双币展示** 以 [2026-03-28-bom-match-currency-mfr-design.md](./2026-03-28-bom-match-currency-mfr-design.md) §1 为准：**单机配置基准币种**、**读取时换算**、**原币与基准币并列展示**、**排序仅使用换算后的可比单价**；税/费口径见该文档 §1.6，多字段优先级 §1.7，汇率日期 §1.8，MOQ/阶梯 §1.9，舍入与平局 §1.10。**`price_tiers` 等阶梯字符串** 默认按该文档 **§1.11 解析**；解析失败或未识别格式时保守回退（§1.5）。
