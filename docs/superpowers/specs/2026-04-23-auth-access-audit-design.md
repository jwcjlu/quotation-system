# 用户鉴权、角色访问控制与操作审计设计

## 1. 目标

为现有 BOM Web 系统补齐一套最小可用的账号体系，解决三个问题：

- 支持普通用户自助注册与登录。
- 按角色控制页面与 API 访问范围。
- 对新增、修改、删除类操作记录审计日志，满足“谁在什么时候改了什么”的可追溯要求。

本次范围明确包含：

- 匿名仅可访问“使用指南”。
- 普通用户仅可访问 `BOM会话`、`匹配单`、`HS型号解析`。
- 管理员可访问全部页面与全部管理能力。
- 管理员账号只能由现有管理员创建。

本次范围明确不包含：

- 找回密码、短信/邮件验证码、单点登录。
- 复杂组织/部门/数据隔离模型。
- 审计日志后台查询页面。

## 2. 当前现状

- 前端 `web/src/App.tsx` 当前直接暴露全部业务入口，没有登录态与角色态。
- 后端当前没有用户、会话、角色三类模型，也没有通用鉴权中间件。
- 现有写接口主要集中在 `internal/service`：
  - `BomService`：BOM 会话、BOM 行的新增/修改/删除。
  - `HsMetaService`：HS 元数据新增/修改/删除。
  - `AgentAdminService`：Agent 脚本凭据、BOM 平台配置新增/修改/删除。
  - `ScriptPackageAdmin`：脚本包上传/发布。
  - `HsResolve` 手工 datasheet 上传：会形成新记录，也需要计入审计。

## 3. 方案选择

本次采用“服务端会话表 + Bearer Token”方案：

- 登录成功后，后端生成随机 session token。
- 前端将 token 保存到本地存储，并在 API 请求头中附带 `Authorization: Bearer <token>`。
- 后端只保存 token 哈希值，不保存明文 token。
- 每次请求通过会话表解析当前登录用户与角色。

采用该方案的原因：

- 与当前前后端 API 调用方式兼容，改造成本低。
- 比纯 JWT 更容易做会话撤销、过期控制与权限变更即时生效。
- 比传统 Cookie Session 更贴合当前前端 `fetch` 调用结构。

## 4. 角色与访问边界

系统定义两类角色：

- `user`：普通用户。
- `admin`：管理员。

访问规则如下：

- 匿名用户：
  - 仅可访问“使用指南”页面与注册/登录接口。
- 普通用户：
  - 可访问 `BOM会话`、`匹配单`、`HS型号解析`。
  - 不可访问 `脚本包`、`Agent运维`、`HS元数据`。
- 管理员：
  - 可访问全部页面与全部 API。

账号创建规则如下：

- `POST /api/v1/auth/register` 只能创建普通用户。
- `admin` 账号只能由已登录管理员通过后台接口创建。
- 任何客户端传参都不能绕过服务端角色约束直接注册管理员。

## 5. 后端接口设计

新增认证接口：

- `POST /api/v1/auth/register`
  - 入参：用户名、显示名、密码。
  - 行为：创建普通用户。
- `POST /api/v1/auth/login`
  - 入参：用户名、密码。
  - 返回：session token、用户资料、角色。
- `GET /api/v1/auth/me`
  - 行为：返回当前登录用户与角色，用于前端刷新恢复登录态。
- `POST /api/v1/auth/logout`
  - 行为：作废当前会话。
- `POST /api/v1/auth/admin/users`
  - 仅管理员可调用。
  - 用于创建管理员账号或普通账号。

鉴权与授权行为：

- 在 `internal/server` 增加通用鉴权中间件。
- 中间件负责：
  - 解析 bearer token。
  - 读取当前用户。
  - 将用户身份写入请求上下文。
- 在 service 或路由层声明接口角色要求：
  - 未登录返回 `401 Unauthorized`。
  - 已登录但角色不足返回 `403 Forbidden`。

## 6. 数据模型设计

新增三张表：

### 6.1 `t_user`

字段建议：

- `id`
- `username`
- `display_name`
- `password_hash`
- `role`
- `status`
- `last_login_at`
- `created_at`
- `updated_at`

约束：

- `username` 唯一。
- `role` 仅允许 `user` / `admin`。
- 密码只保存强哈希，不保存明文。

### 6.2 `t_user_session`

字段建议：

- `id`
- `user_id`
- `token_hash`
- `expires_at`
- `last_seen_at`
- `created_at`

约束：

- `token_hash` 唯一。
- 登出时删除或失效当前会话记录。
- 角色变更后，新请求始终以数据库最新用户角色为准。

### 6.3 `t_audit_log`

字段建议：

- `id`
- `actor_user_id`
- `actor_username`
- `action`
- `resource_type`
- `resource_id`
- `summary`
- `detail_json`
- `created_at`

说明：

- `detail_json` 仅保存关键字段的 before/after 摘要，不复制完整业务大对象。
- 审计记录按写操作成功后落库，不在 data 层做业务决策。

## 7. 审计范围

首版要求所有新增、修改、删除动作必须留痕。

首批覆盖资源：

- `bom_session`
- `bom_session_line`
- `hs_meta`
- `agent_script_auth`
- `bom_platform_script`
- `script_package`
- `user`（管理员创建账号）
- `hs_manual_datasheet_upload`

建议同时记录以下身份事件：

- `login`
- `logout`

首版不要求实现审计日志查询后台，但表结构与落库字段必须为后续查询留够信息。

## 8. 前端交互设计

导航按身份动态裁剪：

- 未登录：仅显示“使用指南”和“登录 / 注册”入口。
- 普通用户：显示 `BOM会话`、`匹配单`、`HS型号解析`、`使用指南`。
- 管理员：显示全部导航。

登录态流程：

- 应用启动时调用 `GET /api/v1/auth/me`。
- 未登录时默认进入“使用指南”。
- 若用户尝试访问受限页面，跳转到登录页或展示登录面板。
- 如果当前角色无权访问当前页面，自动回退到第一个可访问页面。

请求行为：

- 在前端统一请求封装中自动附加 bearer token。
- 收到 `401` 时清空本地登录态并跳回登录入口。
- 收到 `403` 时展示“无权限访问”提示。

管理员能力：

- 在管理员可见页面中新增账号管理入口。
- 管理员可创建新的管理员账号。
- 公共注册页不暴露角色选择。

## 9. 初始化与迁移

数据库迁移：

- 在 [docs/schema/migrations](/D:/workspace/caichip/docs/schema/migrations) 新增用户、会话、审计三张表的 migration SQL。
- 在 `internal/data/models.go` 增加对应 GORM 模型。
- 在 `internal/data/migrate.go` 将三张表纳入自动迁移。

初始化策略：

- 如果数据库内不存在管理员账号，允许通过配置或环境变量注入一个引导管理员。
- 引导管理员仅用于系统首次初始化。
- 初始化完成后，后续管理员账号统一由现有管理员创建。

## 10. 测试策略

后端测试重点：

- 普通注册只能创建 `user`。
- 普通用户不能创建管理员。
- 管理员可以创建管理员。
- 未登录访问受限接口返回 `401`。
- 普通用户访问管理员接口返回 `403`。
- 关键写接口成功后写入审计日志。

前端测试重点：

- 未登录时只显示“使用指南”与登录入口。
- 普通用户登录后仅显示三项业务入口。
- 管理员登录后显示全部入口。
- `401` 会清理登录态并回到登录入口。
- `403` 会显示权限不足提示。

## 11. 实施边界与拆分建议

该需求可拆成四个可独立验证的实现块：

1. 用户、会话、审计的数据模型与基础 repo。
2. 后端认证接口与鉴权/角色中间件。
3. 前端登录态与导航门禁。
4. 现有写接口接入审计日志。

这样可以在不破坏现有业务功能的前提下逐步切换到受控访问模式。

## 12. 风险与约束

- 当前仓库已有较多未提交改动，实施时必须避免覆盖非本次需求变更。
- `internal/service/bom_service.go` 已偏长，新增鉴权与审计接入时应优先拆分辅助函数，避免进一步膨胀。
- 角色校验不能只依赖前端导航裁剪，必须以后端授权为最终准绳。
- 审计日志应在业务成功后记录，失败操作不应伪造成已完成审计。

## 13. 验收标准

当以下条件全部满足时，视为本设计完成：

- 未登录用户只能访问“使用指南”。
- 普通用户登录后仅能使用 `BOM会话`、`匹配单`、`HS型号解析`。
- 管理员登录后可以访问全部页面。
- 管理员能创建管理员账号，普通注册无法产生管理员。
- 关键新增/修改/删除操作均能落审计日志。
- 前后端都能正确处理 `401` 与 `403`。
