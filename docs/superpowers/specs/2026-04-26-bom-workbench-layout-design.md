# BOM 工作台布局优化设计
日期：2026-04-26

## 背景

当前 Web 端把 `BOM会话` 与 `匹配单` 放在两个一级导航 tab 中。`BOM会话` 页负责上传、筛选、打开会话详情；会话详情又通过弹窗承载导入状态、BOM 行、搜索任务、缺口方案、维护信息等二级 tab。`匹配单` 则是另一个一级页面，依赖最近一次选中的 `bom_id`，并要求会话状态达到 `data_ready`。

这种结构功能上可用，但用户心智被拆成了两个入口：

- 先在 `BOM会话` 中找到或修复会话。
- 再跳到 `匹配单` 查看结果。
- 如果发现搜索任务、缺口、会话状态有问题，又回到会话弹窗处理。

布局优化目标是把“围绕一个 BOM 会话完成报价配单”的流程聚合为一个工作台，减少跨页跳转和弹窗嵌套。

## 目标

- 将顶部一级导航中的 `BOM会话` 与 `匹配单` 合并为 `BOM工作台`。
- 在工作台内以“左侧会话列表 + 右侧会话工作区”的布局承载主要流程。
- 右侧工作区用二级 tab 表达会话处理流程：`总览`、`BOM行`、`搜索任务`、`缺口方案`、`匹配单`、`维护信息`。
- `匹配单` 不再依赖单独一级页面，而是成为当前会话的一个工作区 tab。
- 保留 `data_ready` 校验：未就绪时禁用或提示 `匹配单` tab，并在同页说明阻塞原因。
- 移动端保持可用：先显示会话列表，进入会话后显示工作区，并提供返回列表入口。
- 借布局调整拆分现有大组件，避免继续扩大单文件职责。

## 非目标

- 不改变后端接口语义。
- 不改变匹配算法、缺口处理规则、搜索任务重试逻辑。
- 不新增审批流、权限模型或新的报价状态机。
- 不重做全站视觉风格，只在 BOM 工作台范围内优化信息架构和页面密度。

## 推荐方案

采用“BOM 工作台”方案：顶部保留一个 `BOM工作台` 一级入口，进入后左侧是会话列表和筛选，右侧是选中会话的完整工作区。

选择该方案的原因：

- BOM 会话和匹配单本质上是同一条业务链路，不适合作为两个并列系统入口。
- 当前 `匹配单` 已经强依赖最近会话 ID，独立页面反而隐藏了上下文。
- 搜索任务、缺口、匹配结果之间需要频繁切换，放在同一工作区更符合报价员操作路径。
- 左侧列表让用户可以快速切换会话，右侧工作区保留当前处理位置。

备选方案 A 是保留两个一级 tab 但增强上下文，改动小但跳转仍存在。备选方案 C 是以匹配单为主并加右侧健康栏，适合只看最终结果，但前置修复流程仍被拆开。因此本次不采用 A/C。

## 信息架构

### 顶部导航

调整前：

- `BOM会话`
- `匹配单`
- `HS型号解析`
- 其他管理入口

调整后：

- `BOM工作台`
- `HS型号解析`
- 其他管理入口

`BOM工作台` 对匿名、普通用户、管理员均沿用原 `BOM会话` 可见性。原 `匹配单` 的权限能力合并进工作台内的 `匹配单` tab；没有匹配权限时隐藏或禁用该 tab。

### 工作台布局

桌面端使用双栏：

- 左栏：会话筛选、创建按钮、会话列表、分页。
- 右栏：当前会话标题、状态摘要、主要操作按钮、二级 tab 和 tab 内容。

右栏二级 tab：

- `总览`：导入状态、行数、搜索异常数、缺口数、最近匹配方案摘要。
- `BOM行`：BOM 行列表、异常行优先排序、行增删改。
- `搜索任务`：搜索任务覆盖、失败/缺失/可重试任务、批量重试。
- `缺口方案`：缺口列表、人工报价、替代料、已保存方案版本。
- `匹配单`：当前会话的匹配结果、筛选、列配置、HS 跳转、缓存明细。
- `维护信息`：会话标题、客户信息、平台勾选等低频维护字段。

## 关键交互

### 选择会话

用户在左侧点击会话后：

1. 写入当前会话 ID 到页面状态和本地存储。
2. 右侧加载该会话工作区。
3. 默认进入系统推荐 tab。

推荐 tab 规则沿用现有会话页逻辑并调整优先级：

1. 存在搜索异常或可重试任务：进入 `搜索任务`。
2. 存在待处理缺口：进入 `缺口方案`。
3. 存在阻塞可配单的 BOM 行：进入 `BOM行`。
4. 会话已 `data_ready` 且用户从“查看匹配单”动作进入：进入 `匹配单`。
5. 其他情况进入 `总览`。

用户手动切换 tab 后，本次会话内尊重手动选择，不再自动跳转。

### 创建会话

`新建 BOM 单` 仍从工作台左上进入。可以先保留弹窗上传；上传成功后自动选中新会话并在右侧展示导入状态。后续可再将上传也改为右侧空状态，但不作为本次必做。

### 匹配单入口

`匹配单` tab 的状态：

- `data_ready`：可点击，展示匹配结果。
- `parsing` 或其他未就绪状态：禁用，旁边显示“导入/搜索未完成”提示。
- 缺少当前会话：右侧显示空状态，提示先选择或新建 BOM 会话。

原本从会话详情顶部进入 `匹配单` 的按钮改为切换右侧 `匹配单` tab，不再切到独立一级页面。

### 移动端

移动端不使用并排双栏：

- 默认显示会话列表。
- 点击会话后进入会话工作区。
- 工作区顶部提供“返回会话列表”按钮。
- 二级 tab 横向滚动，内容保持单列。

## 组件边界

为避免继续扩大现有页面文件，实施时拆分为以下组件：

- `BomWorkbenchPage`：工作台容器，管理选中会话 ID、左右布局、移动端列表/详情切换。
- `SessionListPanel`：筛选、创建按钮、列表、分页。
- `SessionWorkspace`：当前会话头部、状态摘要、二级 tab、刷新协调。
- `SessionOverviewPanel`：总览指标和导入状态摘要。
- `SessionLinesPanel`：BOM 行管理。
- `SessionSearchTasksPanel`：现有 `SearchTaskStatusPanel` 的承载层。
- `SessionGapsPanel`：缺口处理和方案版本。
- `SessionMaintenancePanel`：单据信息和平台勾选。
- `MatchResultWorkspace`：从 `MatchResultPage` 中抽出可嵌入的匹配单内容。

`App.tsx` 只负责一级导航与权限，不承载 BOM 工作台内部状态。`BomSessionListPage` 可演进为 `BomWorkbenchPage`，但列表、弹窗详情和工作区逻辑应拆开，避免一个文件同时承担列表、弹窗和详情。

## 数据流

工作台选中会话后，右侧工作区加载：

1. `getSession(sessionId)`
2. `getBOMLines(sessionId)`
3. `getSessionSearchTaskCoverage(sessionId)`
4. `listSessionSearchTasks(sessionId)`
5. `listLineGaps(sessionId)`
6. `listMatchRuns(sessionId)`

`匹配单` tab 在激活时加载匹配结果相关数据，避免每次选择会话都立即拉取重表格和缓存明细。保存匹配方案、缺口处理、搜索任务重试等动作完成后，只刷新相关 tab 所需数据，并同步更新总览计数。

## 错误与空状态

- 未选择会话：右侧显示“请选择或新建 BOM 会话”。
- 会话加载失败：右侧显示错误提示并提供重试。
- 匹配单未就绪：显示当前状态、需要完成的步骤、刷新按钮。
- 搜索任务失败：`搜索任务` tab 展示可重试项，左侧会话行可显示轻量异常标记。
- 缺口未处理：`缺口方案` tab 显示数量，`总览` 中给出下一步入口。

## 测试策略

前端测试重点：

- 默认进入 `BOM工作台`，原 `匹配单` 一级导航不再出现。
- 选择会话后右侧加载工作区。
- `data_ready` 前 `匹配单` tab 不可用或展示未就绪提示。
- `data_ready` 后可进入 `匹配单` tab，并使用当前会话 ID 加载结果。
- 搜索异常、缺口、阻塞行存在时推荐 tab 正确。
- 移动端列表/详情切换不丢失选中会话。

不新增后端测试，因为本次设计不改变后端行为。

## 分阶段落地

第一阶段：

- 合并一级导航为 `BOM工作台`。
- 把会话详情从弹窗改为右侧工作区。
- 保留现有上传弹窗。
- 将原 `匹配单` 作为右侧 `匹配单` tab 嵌入。

第二阶段：

- 拆分 `SourcingSessionPage` 和 `MatchResultPage` 内的大块 UI。
- 优化总览指标、推荐 tab、左侧异常标记。
- 完善移动端列表/详情切换。

第三阶段：

- 根据真实使用反馈调整 tab 命名、默认 tab 和按钮位置。
- 评估是否把上传流程也从弹窗改为工作区空状态。
