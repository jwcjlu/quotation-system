# BOM 厂牌清洗：两阶段评审 — 产品需求

## 1. 背景与问题

当前「数据清洗」将 **需求行（`t_bom_session_line`）** 与 **报价明细（`t_bom_quote_item`）** 的厂牌待办混在同一列表，且一次操作可能同时影响两张表，评审职责不清、易混淆。

本需求将厂牌清洗拆为 **先后两阶段**：先定需求侧规范厂牌，再对报价侧做 **仅通过/不通过** 的确认。

**前置说明**：产品尚未正式上线，**不要求**对历史接口或旧交互做兼容；以实现职责清晰、模型简单为优先。

**关联文档**：

- 技术方案：`2026-05-04-bom-mfr-cleaning-two-phase-design.md`
- **软件需求规格（SRS，实现/验收）**：`2026-05-04-bom-mfr-cleaning-two-phase-software-requirements-spec.md`
- **开发计划**：`docs/superpowers/plans/2026-05-04-bom-mfr-two-phase-implementation.md`（文末 **「收尾状态（Phase 6）」** 汇总废弃接口从代码中移除的核对结论）
- 厂牌字段与别名总述（部分内容由本需求替代演进）：`2026-04-18-manufacturer-canonicalization-design.md`

---

## 2. 目标

1. **阶段一（需求行）**：仅针对 `t_bom_session_line`，由运营/用户选定或新建需求侧规范厂牌 `manufacturer_canonical_id`（与 `t_bom_manufacturer_alias` 规则一致）。
2. **阶段二（报价明细）**：仅在需求行规范已就绪后开放；对 `t_bom_quote_item` **仅允许「通过 / 不通过」**；界面 **只读展示** 对应需求行的 `manufacturer_canonical_id`，**不提供** canonical 下拉或手写报价侧 canonical。
3. 拆掉「同一屏混合需求与报价、同一动作双表回填」的产品体验。

---

## 3. 用户角色与场景

- **主要角色**：使用 BOM 工作台「数据清洗」完成会话内厂牌治理的运营/采购用户。
- **典型路径**：导入 BOM → 搜索有结果 → 进入数据清洗 → **先完成需求行厂牌** → **再处理报价厂牌通过性** → 再进入配单/缺口等后续流程（与现网就绪规则衔接处见技术方案）。

---

## 4. 功能需求

### 4.1 阶段一：需求行厂牌清洗

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R1.1 | 列表仅展示「需要清洗」的需求行：需求厂牌原文非空且尚未具备有效 `manufacturer_canonical_id`（具体判定与后端一致） | 列表中不出现纯报价维度条目 |
| R1.2 | 用户可从标准厂牌列表选择 canonical，或按规则手动新增别名（写入别名表） | 与现网别名写入校验一致（如 `alias_norm` 唯一等） |
| R1.3 | 提交后仅更新本条（或明确策略下的）需求行及别名表，**不**在本阶段写报价明细的 canonical | 回归用例可断言 quote 表无连带写入 |
| R1.4 | 支持「应用已有别名」类批量能力时，**仅作用于需求行**（不与报价阶段混写） | 产品说明与按钮文案一致 |

### 4.2 阶段闸门

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R2.1 | 当会话内 **所有需要规范厂牌的需求行** 均已具备 `manufacturer_canonical_id` 后，才允许进入阶段二 | 未满足时阶段二区域禁用或列表为空并提示原因 |
| R2.2 | 「需要规范厂牌的需求行」**不包含**「需求厂牌 `mfr` 为空」的行（见 §5） | 空 `mfr` 行不阻塞闸门 |

### 4.3 阶段二：报价厂牌通过 / 不通过

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R3.1 | 列表仅展示：父需求行已有 `manufacturer_canonical_id`，且本条报价按规则仍「待评审」的 `t_bom_quote_item` | 与后端入队规则一致 |
| R3.2 | 每条展示：报价厂牌原文、平台/型号/封装等上下文；**只读**展示父行的 `manufacturer_canonical_id`（可读展示名若有） | 无 canonical 下拉 |
| R3.3 | 用户操作仅为 **通过** 或 **不通过**；可选填不通过原因 | 无二值以外的主操作 |
| R3.4 | **通过**：表示承认该报价厂牌与当前需求行规范一致，系统落库规则见技术方案（报价侧 canonical 与父行一致） | 配单侧可按「已确认」处理 |
| R3.5 | **不通过**：表示该条报价在厂牌维度不可用，系统落库规则见技术方案 | 配单侧排除或进缺口，与产品统一口径 |

### 4.4 改判

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R4.1 | 允许对同一报价明细 **重复提交** 评审结果（改判） | 至少支持：不通过 → 通过 |
| R4.2 | 允许 **通过 → 不通过**（若业务需要收回承认） | 与 R3.4/R3.5 落库一致、无脏状态 |

## 5. 业务规则（与产品已确认口径）

### 5.1 需求厂牌 `mfr` 为空的行

- **不参与**阶段一待清洗列表，**不要求**填写 `manufacturer_canonical_id`。
- **不阻塞**阶段闸门：视为该行在「需求侧厂牌」维度已就绪。
- **其下所有报价**在厂牌评审维度 **默认视为已通过**：不要求用户进入阶段二对该行下报价做通过/不通过；读模型与配单规则按「无需求厂牌约束」处理（与 `2026-04-18-manufacturer-canonicalization-design.md` 中「manufacturer 为空不施加厂牌约束」一致）。
- 若需 **审计可追溯**，可由技术方案选择是否批量写入 `manufacturer_review_status=accepted`；**不**强制用户逐条点通过。

### 5.2 未上线策略

- **不保留**旧版「需求+报价混合列表 + 单次确认同时影响两行表」的产品路径作为正式能力；以本需求与对应技术方案为唯一口径。

---

## 6. 非功能需求（摘要）

- **一致性**：同一会话内，**通过**后的报价明细在厂牌维度与父行 canonical 一致，避免双轨 canonical。
- **可说明性**：闸门未开、列表为空、不通过原因等须有用户可理解文案。
- **权限**：与现网 BOM 会话编辑权限一致（若后续分角色再单开文档）。

---

## 7. 前端信息架构（摘要）

- 「数据清洗」内 **先后两块**：「需求行厂牌」→「报价厂牌确认」；闸门未满足时第二块禁用并说明。
- 第二块无 canonical 下拉，仅只读父行 canonical + 通过/不通过（及原因）。

---

## 8. 开放项（若后续要补）

- **不通过**是否自动写入缺口表、是否触发重搜：由配单/缺口产品统一，本需求只要求「配单侧可区分未审/通过/不通过」。
- 标准厂牌列表与别名冲突提示：沿用现网交互即可。
