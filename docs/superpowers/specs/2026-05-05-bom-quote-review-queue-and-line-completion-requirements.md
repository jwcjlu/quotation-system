## 1. 背景与问题

在多平台搜索同一需求行（`t_bom_session_line`）时，会产生大量 `t_bom_quote_item`（报价行）。即便订单规模不大，**待审报价行条数**仍可能达到数百至上千，运营在「厂牌/报价确认」上耗时过长。

**业务共识**：

- 配单逻辑上，在**型号一致**且**其它可比参数均满足约束**的前提下，倾向**价格最优**。
- 不依赖大模型做自动通过/拒绝；以**确定性规则 + 人工在收窄集合上确认**为主。

## 2. 目标与非目标

### 2.1 目标

| 编号 | 描述 |
|------|------|
| G1 | **可控审核量**：默认只把「最值得先看」的报价行推入优先队列（低价 TopN，默认 N=5，可配置）。 |
| G2 | **不丢数据**：全量报价行仍保留在库中；默认队列仅为排序与展示优先级，须支持全量展开与改判。 |
| G3 | **可执行的结案规则**：对单条需求行给出**是否允许结束本行报价评审阶段（规则 B）**的明确布尔语义（详见 SRS）。 |
| G4 | **与厂牌两阶段对齐**：候选比价池排除 `rejected`；与 `manufacturer_review_status` 语义一致。 |

### 2.2 「结案」产品语义（须与运营对齐）

规则 B 表示：在**已进入比价候选池**的报价中，**当前底价带（最多 3 条）在厂牌维度已全部通过**，从而不存在「池内更便宜但仍未决」的结构性漏审。

**不表示**：本需求行下**每一条**报价（含未进池者）均已人工处理完毕。若产品需要「行级全清完成」，须**另立字段或另立规则**，不得与规则 B 混用同一布尔而不加说明。

### 2.3 非目标（本期）

- 具体 API、事务边界、与 `data_ready` 闸门的耦合细节（见 SRS **§8** REQ-READY-* + 实现计划）。
- 各平台原始字段到 `compare_price` 的映射细则（SRS 只约束口径与缺失策略选项）。
- 厂牌以外的独立审批维度（如渠道合规）的多维池交集（另文）。

## 3. 用户角色与场景

- **角色**：使用 BOM 工作台处理报价厂牌评审的运营/采购。
- **场景**：在大量报价中**优先看到低价带附近**的条目；在规则满足时，系统可判定本行**报价评审维度可结案**（与上层会话/配单闸门衔接由 SRS/实现计划约定）。

## 4. 功能需求

### 4.1 默认审核队列（TopN）

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R1.1 | 对单条需求行，在「仍参与比价」的集合上按可比价升序取前 **N** 条（默认 **5**），作为工作台**默认优先展示/推送**的审核集合 | N 可配置；未进 TopN 的报价仍可被查、可改判 |
| R1.2 | TopN **仅为 UI/队列优先级**，不表示未进 TopN 的报价被删除或否决 | 提供一键「全量」或等价筛选入口 |
| R1.3 | 工作台须区分 **TopN（先审哪几条）** 与 **影响结案的底价带 TopK（最多 3 条）**，避免「审完 5 条即以为可结案」的误解 | 对 TopK 条有可见标识（命名/色块/列等，细节见 SRS REQ-UI-*） |

### 4.2 需求行结案（规则 B）

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R2.1 | 系统 SHALL 按 SRS 定义计算「本行是否满足规则 B」；**无候选（m=0）时不得**因空集而误判为可结案 | 见 SRS REQ-RULE-B-001 |
| R2.2 | 当 **m≥1** 时，规则 B 等价于：按 SRS 排序后的 **TopK（K=min(3,m)）** 内每一条厂牌评审状态均为 `accepted`，且可选辅助条件 B-aux-1 / B-aux-2 按评审选定写死 | 与 SRS 公式一致 |
| R2.3 | `rejected` 报价不参与候选池比价与 TopK 重算；改判后系统重新计算池与结案布尔 | 见 SRS REQ-POOL-* |
| R2.4 | 新报价进入、价格或参与比价资格变化时，系统重新计算结案条件；是否允许「已结案」回退由产品选定（开放项） | 见 SRS REQ-RECALC-* |

### 4.3 可比价缺失（产品二选一）

| 编号 | 需求描述 | 验收要点 |
|------|----------|----------|
| R3.1 | **M1**：缺失可比价的报价不参与候选池 S，直至补全或标为无效 | 与 SRS 一致；待办展示策略在 SRS 说明 |
| R3.2 | **M2**：缺失可比价则阻塞本行结案并在待办高亮 | 与 SRS 一致 |

（M1/M2 在评审拍板前，实现可按 SRS 占位开关或文档标注「待定」。）

## 5. 建议验收用例（产品层摘要）

完整矩阵与期望见 SRS「验证与验收」；下表便于评审会走读。

| 编号 | 场景 | 期望 |
|------|------|------|
| AC-1 | S 共 3 条，价格序 10/20/30，均为 pending | 不满足结案；全转 accepted 后满足 |
| AC-2 | S 共 5 条，最便宜 3 条中 1 条 pending | 不满足；该条 accepted 或 reject 移出池后重算 |
| AC-3 | 最便宜一条 rejected | 不参与 S；TopK 在剩余集合上重算 |
| AC-4 | 同价并列 | TopK 由稳定排序唯一确定 |
| AC-5 | m=0 | 不满足结案；不得因空 TopK 通过 |
| AC-6 | 最便宜 K 条均 accepted，下方仍有大量 pending | 满足规则 B（与 §2.2「非行级全清」一致） |

## 7. 评审待定清单（产品拍板）

1. **B-aux-1 vs B-aux-2**：是否坚持「至少 3 条 accepted」；与 `|S|<3` 的例外句。
2. **M1 vs M2**：可比价缺失策略。
3. **结案后新报价**：是否允许回退「已结案」；不允许时的锁策略。
4. **TopN 是否收窄 S**：维持全池还是「仅 TopN 参与结案」。
5. **|S|=0 时上层行为**：是否接受规则 B 恒假直至有候选；是否需「无可用报价—人工完结」终态。
6. **§2.2 结案语义**：是否另需「行级全清」字段/规则。
7. **报价过期 / 移出 E**：是否与结案重算联动。
8. **TopK 同价多样性**：是否去重/分层取 TopK。
