# 分布式采集 Agent 需求与架构

本文档整理 **caichip 服务端** 与 **部署在各主机上的 Agent** 之间的协作需求，用于指导后续协议设计与开发。

**接口契约（JSON 路径与字段）**：见 **[分布式采集Agent-API协议](./分布式采集Agent-API协议.md)**（双心跳、任务结果上报、脚本下载）。

---

## 1. 背景与目标

- **目标**：在多台异构主机上部署轻量 Agent，统一由 **caichip 服务** 管理脚本版本与任务调度，实现脚本同步、环境准备与任务执行闭环。
- **约束**：Agent 可分布在不同网络/主机；与服务端通过 **HTTPS + REST 长轮询**（周期性请求，见 §4.2、§6.1、§7）保持可达；需可观测（心跳、版本、任务状态）。**心跳分两类**：任务心跳（默认 10s）与脚本安装心跳（默认 10min），见 §4.2。
- **脚本形态（当前约定）**：业务脚本以 **Python** 为主；caichip 下发的 **脚本安装包为 ZIP**，至少包含 Python 源码与 **`version.txt`**；**`requirements.txt` 可选**（缺失视为无依赖，见 §2）（见下文「脚本安装包规范」）。

---

## 2. 脚本安装包规范（Python / ZIP）

caichip 与 Agent 之间以 **ZIP 压缩包** 作为脚本发布单元，典型目录结构如下（文件名可约定，需在协议或元数据中固定）：

| 文件/目录 | 说明 |
|-----------|------|
| `*.py` | Python 脚本（可多级目录）。 |
| `requirements.txt` | **依赖声明**（**可选**）。若存在，Agent 执行 `pip install -r requirements.txt`；**若缺失，视为无额外依赖**（不判为校验失败，见 §4.4）。 |
| `version.txt` | **版本号**（单行文本），与心跳上报、服务端比对一致；**格式见 §6.5（允许可选 `v` 前缀）**。 |
| 其他 | 可选：配置模板、`README`、数据文件等（以安全策略允许为准）。 |

**版本一致性**：解压后读取 `version.txt` 的文本，经 **规范化**（去首尾空白；**允许** 可选前缀 **`v`/`V`**，比对时与无 `v` 的同一语义版本 **等价**，如 `v1.2.0` ≡ `1.2.0`）后，应与 caichip 下发的 `script_id` + **目标版本**一致；不一致则标记为校验失败，不上报为「已就绪」。

**入口脚本（已约定：二者均支持）**：任务执行时需定位「要运行哪一个 `.py`」，**两种方式同时支持**：

- **约定默认入口**：脚本包根目录下按固定优先级查找存在的文件，例如 **`main.py` → `run.py`**（顺序以协议/OpenAPI 为准）。  
- **任务载荷指定**：任务中可带 **`entry_file`**（相对包根的路径，如 `subdir/job.py`）及可选 **`argv`**；**若任务携带 `entry_file`，则优先于默认约定**。

二者不互斥：未指定 `entry_file` 时使用默认入口；指定则执行对应文件。

**任务载荷中的 `version`（已约定）**：下发任务中的 **`version` 即** 该任务所要使用的 **本地已就绪脚本包版本**——与本地目录 `script_id`/包内 **`version.txt` 规范化后的字符串须一致**（与 §2「版本一致性」同一套规范化规则）；**不是**独立于本机包的“目标版本号”另行解释。

## 3. 总体架构（概念）

```
┌─────────────────────────────────────────────────────────────┐
│                     caichip 服务                              │
│  · 脚本仓库（脚本 ID、版本、校验和）                          │
│  · Agent 注册表（主机标识、在线状态、已同步脚本清单）         │
│  · 任务队列与调度（标签、队列、能力位=已就绪 script_id，见 §4.6）│
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS + REST 长轮询（见 §4.2、§6.1、§7）
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   ┌─────────┐         ┌─────────┐         ┌─────────┐
   │ Agent A │         │ Agent B │         │ Agent C │
   │ 主机 1   │         │ 主机 2   │         │ 主机 N   │
   └─────────┘         └─────────┘         └─────────┘
```

## 4. 功能需求

### 4.1 多主机部署（需求 1）

| 项 | 说明 |
|----|------|
| **部署形态** | Agent 独立进程/服务，可安装在 Windows / Linux 等不同主机。 |
| **身份** | 每个 Agent 具备全局唯一标识 **`agent_id`，约定为本机 MAC 地址**（取主网卡或配置指定的网卡，规范化字符串，格式以 OpenAPI 为准，如小写无分隔 12 位十六进制或 `aa:bb:cc:dd:ee:ff`）。**可选**：若存在多网卡冲突或虚拟化导致重复，允许在配置中 **显式覆盖** `agent_id`。 |
| **发现** | caichip 以 `agent_id`（MAC）区分节点，不依赖固定 IP；支持后续扩容与替换主机。更换网卡或覆盖配置后 `agent_id` 变化，服务端视为 **新节点**（若需沿用旧身份，应通过配置固定原 `agent_id`）。 |
| **生产建议（已约定）** | 云主机、容器、克隆镜像等场景下 MAC **易变或重复**，**生产环境推荐**在配置中 **显式固定 `agent_id`**（或机器指纹方案），**勿仅依赖**默认 MAC 推导。 |

**交付物建议**：安装包或脚本、配置文件模板、 systemd / Windows 服务说明。

### 4.1.1 Agent 启动与 Python / pip 环境自检（已约定）

业务脚本依赖 **Python 解释器** 与 **`pip` 工具**（用于安装 `requirements.txt`）；Agent **启动时** 须先保证 **Python 可用**，再保证 **pip 可用**（缺失则自举安装），然后进入鉴权与双心跳等逻辑。

| 项 | 说明 |
|----|------|
| **启动检查** | Agent 启动后 **首先检测** 是否存在 **满足版本要求** 的 Python（下限如 `>=3.10`，以 Agent 配置或发布说明为准）。检测对象可为：环境变量 `PATH` 中的 `python`/`python3`，或配置项 **`python_executable`** 指向的绝对路径；需能执行 `-V` 并得到合规版本。 |
| **缺失时：自举安装** | 若 **未检测到** 合规解释器，Agent **应自行完成 Python 安装**（Bootstrap），而非仅报错退出（除非配置为「禁止自动安装」）。典型策略（**以实现与平台为准**）：Windows 使用官方安装包静默安装、embeddable 解压到 Agent 数据目录，或企业提供的离线包；Linux 在具备权限时调用 `apt`/`dnf`/`yum` 等，或解压官方二进制到 Agent 私有目录。安装目标路径建议固定（如 `<AgentData>/runtime/python`），便于后续 venv 引用。**企业环境**常禁止自动安装解释器：部署手册应要求 **预装 Python** 或开启「禁止自举」并走 **告警/受限心跳**（见「失败处理」）。 |
| **校验** | 自举完成后再次检测版本；可校验安装包 **SHA-256**（从 **可配置的可信镜像** 下载，避免供应链风险）。 |
| **失败处理** | 若自检与自举均失败：写入本地状态（如 `runtime_error`），**不进入** 任务心跳/脚本同步心跳的正常循环（或仅打受限「告警心跳」）；日志中明确提示需管理员手动安装 Python 或开放权限。 |
| **与脚本包关系** | §4.4 中创建 venv、执行 `pip install` 时，**统一使用** 本节自检通过的解释器；**pip 工具**须按下文「pip 工具自检与安装」就绪后，再安装各脚本包依赖。 |

**pip 工具自检与安装（已约定）**

脚本依赖安装依赖 **`pip`**。在 Python 解释器可用后，Agent **还须** 保证 `pip` 可用；若缺失，**由 Agent 自行安装**，不得假设系统已预装。

| 项 | 说明 |
|----|------|
| **检查** | 使用已就绪的解释器执行 **`python -m pip --version`**（或等价检测），确认 `pip` 可调用。 |
| **缺失时自举** | 若不可用，Agent **须自动完成 pip 安装**，例如（**按顺序尝试，以实现为准**）：① **`python -m ensurepip --upgrade`**（标准库引导）；② 下载官方 **`get-pip.py`**（从 **可配置可信 URL**，校验 **SHA-256**）后执行；③ Windows **embeddable** 等精简发行版按官方文档单独部署 pip。安装过程可复用 §4.4「pip 镜像源与代理」相关配置。 |
| **失败处理** | 与 Python 自举失败策略一致：记录错误，**不进入** 正常双心跳及脚本 `env_status=ready` 路径，直至 pip 可用或人工修复。 |
| **与 §4.4 关系** | 创建 venv 后，venv 内 `pip` 若仍异常，可再次在本机 Python 上引导或修复，再执行 **`pip install -r requirements.txt`**。 |

### 4.2 连接 caichip 并保持心跳（需求 2）

Agent 与 caichip 之间约定 **两种心跳包**，职责与周期 **分离**，避免高频任务轮询与低频脚本同步相互干扰。

| 心跳类型 | 默认周期 | 主要用途 |
|----------|----------|----------|
| **任务心跳** | **每 10 秒** | 维持任务通道在线、上报轻量状态（含 `installed_scripts` 等，见 API 协议）；**在响应中捎带待执行任务**（见 §4.5）；任务执行完成后通过 **任务结果上报接口**（见 **[分布式采集Agent-API协议](./分布式采集Agent-API协议.md)** §4），**不依赖**任务心跳捎带结果。 |
| **脚本安装心跳**（脚本/插件同步） | **每 10 分钟** | 上报 **已同步脚本清单**、**`env_status`**（见 §4.4）；**在响应中返回需下载/升级的脚本包信息**（见 §4.3）。 |

**接口划分（职责单一）**：

- **任务心跳** 与 **脚本安装心跳** 使用 **不同的 HTTP 接口（不同路径）**，各自请求/响应模型独立，**不共用同一 URL**、不通过 `heartbeat_type` 等字段在同一路径上复用，避免职责混杂、便于鉴权限流与监控拆分。  
- 具体路径由 OpenAPI/接口文档约定（示例：`POST .../agent/task/heartbeat`、`POST .../agent/script-sync/heartbeat`，仅作命名示意，以最终实现为准）。

**其它说明**：

- 周期 **默认可配置**（任务心跳默认 10s、脚本安装心跳默认 10min），部署时以配置文件为准。  
- **连接**：Agent 启动后分别调用上述两类端点，协议 **已约定为 REST + HTTP 长轮询**（Long Polling）：请求发出后服务端在有新数据或超时前挂起响应再返回；**两类心跳各用独立路径**，与 §4.2「分接口」一致。  
- **容错**：断线重连、指数退避；某一类心跳失败不应阻塞另一类（待细化）。  
- **服务端在线判定（已约定）**：caichip 以 **任务心跳** 为 Agent **主在线信号**；**默认** **连续 120 秒**（2 分钟）无 **成功** 任务心跳则判 **挂起/离线**；恢复心跳后重新视为在线。脚本安装心跳 **不参与** 该阈值。若配置了 **非默认心跳间隔**，离线判定的 **`T_offline_sec`** 见 **§6.1**（与固定 2 分钟取 max）。与 **长轮询、HTTP 超时** 的对齐见 **§6.1**。  
- **执行中仍按时发心跳（已约定）**：任务子进程执行与 **任务心跳** 须 **异步**：**正在执行任务时仍按配置周期** 发起任务心跳（由独立调度/线程/协程承担），**不得** 因同步阻塞于脚本执行而导致在 **`T_offline_sec`** 内无成功任务心跳、被误判为离线。

### 4.3 脚本安装心跳中的同步信息与服务端下发（需求 3）

本小节对应 **脚本安装心跳（默认 10 分钟一次）**。

| 项 | 说明 |
|----|------|
| **Agent → caichip（脚本安装心跳载荷）** | 携带 **当前已同步的脚本清单**：建议包含 `script_id`、`version`（或内容哈希）、**`env_status`**（环境准备状态，见 §4.4）、可选 `installed_at`。 |
| **caichip 侧逻辑** | 对比「服务端权威脚本列表」与 Agent 上报列表，识别 **缺失脚本** 或 **版本落后**。 |
| **caichip → Agent** | 在 **脚本安装心跳接口的响应** 中返回 **待同步项**：脚本元数据 + 下载地址或分块内容 + 版本信息。 |
| **一致性** | 建议对脚本包做哈希校验（SHA-256），避免损坏或篡改。 |

**状态机简述（同步阶段）**：`未安装 → 下载中 → 校验通过 → 环境准备中 → 就绪`；版本升级走同样通道。

### 4.4 脚本落地、基于 requirements.txt 的环境准备与状态记录（需求 4）

| 项 | 说明 |
|----|------|
| **落地** | 将 ZIP 保存到约定目录（如 `script_id/version/`），解压到同级或子目录；校验 ZIP 哈希与 `version.txt`。 |
| **Python 环境** | **已约定：每个脚本包（每个 `script_id` + `version`）使用独立 venv**，路径如 `.../script_id/version/.venv`，**禁止** 多个脚本包共用同一 venv，以避免依赖冲突与版本串扰；使用的基座解释器须为 **§4.1.1 自检通过**（或自举安装成功）的 Python。 |
| **依赖安装** | 若包内 **存在** `requirements.txt`，在 venv 中执行 **`pip install -r requirements.txt`**（或等价命令）；可固定 `pip` 版本与超时、重试策略。**若不存在该文件，视为无依赖**，跳过 pip 安装步骤，不视为校验失败。**前提**：**§4.1.1** 中 Python 与 **pip** 均已就绪；若 venv 内 pip 异常，按 §4.1.1「pip 工具」规则修复后再装依赖。 |
| **准备状态（须持久化并上报）** | Agent 对每个 `script_id + version` 维护 **环境准备状态**，至少包含： |

**pip 镜像源与代理（已支持）**

Agent 执行 `pip install` 时须 **支持配置 PyPI 镜像源** 与 **HTTP(S) 代理**，便于内网或国内网络环境，来源优先级建议：

1. **Agent 本地配置文件**（如 `pip_index_url`、`pip_trusted_host`、`http_proxy`/`https_proxy` 等字段，键名以最终实现为准）；  
2. **进程环境变量**（如 `PIP_INDEX_URL` / `PIP_EXTRA_INDEX_URL`，以及 `HTTP_PROXY`、`HTTPS_PROXY`、`NO_PROXY`）；  
3. 执行 pip 时传入等价参数（如 `pip install -i <镜像URL> -r requirements.txt`，`--proxy` 等）。

**说明**：镜像与代理仅作用于 **依赖安装阶段**，不改变脚本包内容；敏感代理信息仅存于 Agent 主机，**不得** 写入脚本 ZIP。

**环境准备状态（建议字段）**

| 字段 | 说明 |
|------|------|
| `script_id` / `version` | 与 `version.txt`、服务端一致。 |
| `env_status` | 枚举，例如：`pending`（未准备）、`preparing`（安装依赖中）、`ready`（可执行任务）、`failed`（准备失败）。 |
| `message` | 失败时的简要原因（如 pip 报错摘要，注意长度与脱敏）。 |
| `updated_at` | 最后状态变更时间。 |
| `pip_exit_code` | 可选，记录最后一次 `pip install` 退出码。 |

**上报时机**：在 **脚本安装心跳（默认每 10 分钟）** 载荷中携带各脚本的 `env_status`（与「已同步版本」并列），便于 caichip 控制台展示「已下载但未就绪」「依赖安装失败」等；`failed` 时可由服务端决定是否重试下发或告警。若本地刚完成安装/准备，也可 **主动触发一次** 脚本安装心跳以加速同步（可选实现）。

**多版本并存与清理（已约定）**：同一 **`script_id`** 下可并存 **多个 `version/`** 目录；**清理策略**（保留数量、磁盘上限、按年龄删除）见 **§6.7**，避免长期胀盘。

**其他检查**：Python 主版本、**pip 可用性**（见 §4.1.1）、磁盘空间、镜像/代理可达性等可在 `preparing` 前预检，失败同样写入 `env_status=failed`（见上文「pip 镜像源与代理」）。

**安全建议**：仅允许执行已解压且校验通过的路径下的解释器与脚本；禁止执行 ZIP 内任意 shell；敏感配置由 caichip 或主机侧注入环境变量。

### 4.5 任务下发、定位脚本并执行、结果回传（需求 5）

| 项 | 说明 |
|----|------|
| **任务发现** | caichip 在 **有任务需要某 Agent 执行** 时，将任务 **随任务心跳（默认每 10 秒）的长轮询响应一并返回**，与 **脚本安装心跳** 解耦。 |
| **任务载荷** | **必填**：`task_id`、**`script_id` + `version`**。其中 **`version` 即本地已就绪脚本包版本**：须与 **`script_id` 对应目录下 `version.txt` 规范化后一致**（见 §2）；不一致则 **拒绝执行** 并上报（如 `skipped`/`failed`，以 OpenAPI 为准）。**可选**：`entry_file`（相对包根）、参数 JSON、**执行超时 `timeout_sec`（秒）**、优先级、幂等键等。**未提供 `entry_file` 时**，入口为脚本包根目录下 **`main.py` → `run.py`** 顺序查找第一个存在的文件（以 OpenAPI 为准）。**超时**：由 **下发任务** 指定 `timeout_sec`；**若未给出则默认 300 秒（5 分钟）**，见 **§6.2**。 |
| **定位脚本** | Agent 根据 **`script_id`、`version`** 定位到本地目录；确认 **`env_status == ready`** 且 **`version` 与本地包一致** 后：若任务含 **`entry_file`** 则执行该文件；**否则**按 **`main.py` → `run.py`** 默认顺序，使用 **该包 venv 内 Python** 启动。 |
| **执行** | 子进程执行（工作目录建议为脚本包根目录）、资源与超时限制、捕获 stdout/stderr、退出码。**超时由 Agent 执行层计时并强制终止**：达到 **`timeout_sec`（未下发则用默认 5 分钟）** 时，**终止该任务对应的整个进程组**（含子进程；POSIX 进程组 / Windows Job Object 等，以实现为准），**不得** 仅终止主进程而遗留子进程；并上报 **`status=timeout`** 及日志摘要，见 **§6.2**。 |
| **并发与「同类任务」** | **串行互斥粒度 = `script_id`**：同一 Agent 上，**同一 `script_id`（同一种脚本包）同时最多 1 个任务**；不同 `script_id` 在「同步类」前提下可并行；**全局并行上限**与异步/长任务策略待 OpenAPI。**任务类型**（若协议中定义，如同步/异步、短任务/长任务）表示 **执行方式与超时/资源策略**，**不等于**「同类任务」——**同类 = 同一 `script_id`**，二者勿混用。 |
| **结果回传** | 执行结束后 **调用任务结果上报接口** 提交：`task_id`、状态（成功/失败/超时）、日志摘要、结构化 `result`（JSON）；本地文件路径须由 Agent 读入后写入 `result`，见 API 协议 §4.3；**单请求大小与 tail 截断**见 **§6.6**。若 `env_status != ready` 应拒绝执行并上报原因。 |
| **幂等与重复** | 同一 `task_id` 的 **重复结果上报**、**重复执行** 的语义见 **§6.3** 与 API 协议 **§4.2**。 |

**可选增强**：任务取消、进度心跳、大日志对象存储直传。

### 4.6 任务与 Agent 的匹配规则（已约定：标签、队列、能力位）

caichip 在将任务分配给 Agent 时，**同时支持**以下维度（可组合使用，具体过滤逻辑以实现为准）：

| 维度 | 说明 |
|------|------|
| **标签（tags）** | Agent 注册/心跳时上报自身标签（如 `region=cn`、`env=prod`）；任务可携带 **所需标签** 或标签表达式，仅分发给满足条件的 Agent。 |
| **队列（queue）** | 任务与 Agent 可归属 **具名队列**（如 `default`、`priority`、`scraping`），调度仅在 **同队列内** 或按规则跨队列，用于隔离负载与优先级。 |
| **能力位（capabilities）** | **已约定：能力位即本 Agent 已安装且环境就绪的脚本的 `script_id`。** 亦即 **`installed_scripts`（或脚本安装心跳中的 `scripts`）里 `env_status=ready` 的 `script_id` 集合**；任务若绑定某 `script_id`，仅分发给 **具备该脚本且版本匹配** 的 Agent。**不再** 将 `browser`、`gpu` 等与脚本 ID 混作能力位主语义（若需主机环境信息，可用 **标签** 另行表达，如 `has_browser=true`）。 |

**原则**：**标签 + 队列 + 能力位（ready 的 `script_id`）** 可同时用于一次调度决策。任务载荷与 Agent 元数据字段名以 **[分布式采集Agent-API协议](./分布式采集Agent-API协议.md)** 为准。

**与 API 的对应**：**「能力位」不单独维护一套与脚本无关的枚举**；以 **`installed_scripts` / `scripts` 中的 `script_id` + `env_status`** 为准，服务端由 **`env_status=ready` 的 `script_id`（及版本）** 推导可派发的脚本类任务。

**离线任务重派（已约定）**：当某 Agent 被判定 **挂起/离线**（§4.2、§6.1）时，caichip 对 **尚未收到终态结果**（如 `success` / `failed` / `timeout` / `skipped`）的任务执行 **重派**：**重新入队** 或 **改派至其它在线 Agent**（与幂等、`task_id`、`attempt`、**`lease_id`**、迟到结果拒收见 **§6.3、§6.4** 与 API 协议 **§4.2**），避免任务长期悬挂。

### 4.7 与现有 caichip 代码仓库及配置集成（实施约定）

本节约定 **Agent 能力在现有 caichip 服务端中的落位方式**；传输协议见 **§7（REST 长轮询）** 已决项，本节约定 **代码与配置落位**，二者互补；**以当前仓库结构为事实依据**（Go + Kratos v2）。

#### 工程现状（参考实现）

| 项 | 说明 |
|----|------|
| **进程入口** | `cmd/server`：启动时 `-conf` 指向 **YAML 配置文件**（如 `configs/config.yaml`），使用 Kratos `config` + `file.NewSource` 加载。 |
| **配置模型** | `internal/conf` 中 `Bootstrap`：含 `server`（`http`/`grpc` 地址与超时）、`data`（数据库、Redis）、`platform`（各平台爬虫 URL、`crawler_path`、`work_dir` 等）。 |
| **HTTP 服务** | `internal/server/http.go`：创建 Kratos `http.Server`，向其中 **注册** 现有 gRPC-Gateway 风格 API（如 `RegisterBomServiceHTTPServer`），监听 **`server.http.addr`**。 |

#### 服务端集成原则

| 项 | 说明 |
|----|------|
| **同进程、同端口** | Agent 相关 HTTP 接口（任务心跳、脚本同步心跳、任务结果上报、脚本包下载等）在 **同一 `http.Server`** 上 **新增路由**（建议统一前缀，如 `/api/v1/agent/...`），**不**单独再起监听端口，降低运维与防火墙成本。 |
| **路由与中间件** | 在 Kratos HTTP **中间件链** 中对 `/api/v1/agent/**`（或最终 OpenAPI 路径）做 **API Key** 校验；与现有 BOM 等业务路由 **前缀隔离**，避免鉴权策略混淆。 |
| **配置扩展** | 在 `Bootstrap`（或等价结构）中 **新增段**（如 `agent:`），存放：脚本 ZIP **存储根路径**或对象存储连接、**下载 URL 对外前缀**、API Key **校验配置**（密钥或校验表）、任务/脚本元数据相关项等；键名以实现为准。 |
| **与现有 `platform.*` 的关系** | 现有 `platform.*.crawler_path` / `work_dir` 表示 **本机直接执行** 爬虫脚本的路径；**分布式 Agent 模式**下，执行发生在 **Agent 主机**，caichip 侧重 **脚本仓库 + 调度 + 心跳**；两种模式可 **并存**，由业务与部署选择，文档不互斥。 |

#### 配置「中心」与部署

| 项 | 说明 |
|----|------|
| **当前形态** | 以 **本地/挂载 YAML** 为主，通过 **`-conf`** 与容器/进程参数注入；符合「配置中心」一词的 **集中下发** 可通过 **CI/CD 渲染 YAML**、**挂载 Secret**、或 Kratos **环境变量覆盖** 实现。 |
| **演进** | 若引入 **Apollo / Nacos / etcd** 等，可作为 Kratos `config` 的 **额外 `Source`**，与 `file` 组合；**键空间**建议与 `Bootstrap` 扩展字段一致，避免两套配置语义。 |
| **Agent 进程侧** | Agent **独立配置**（如 `agent.yaml` 或环境变量）：`caichip_base_url`、`api_key`、`agent_id` 覆盖、pip/代理等，**不** 与 caichip 服务端 `config.yaml` 混用同一文件，以免部署边界不清。 |

#### 代码分层建议（caichip 仓库内）

- **服务层**：`internal/service` 下新增 Agent 相关 Service（心跳处理、任务领取、结果写入）。  
- **领域层**：`internal/biz` 下任务调度、与 §4.6 标签/队列/能力匹配。  
- **数据层**：`internal/data` 下 Agent 注册、任务状态、脚本版本元数据；**复用** `conf.Data.Database` / Redis（若任务队列用 Redis）。  
- **Wire**：`cmd/server/wire.go` 注入新依赖，保持与现有 `wire` 风格一致。

#### Agent 可执行体位置

- **Python Agent 进程**（或未来 Go Agent）可作为 **独立子工程**（如仓库根目录 `agent/` 或独立仓库 `caichip-agent`），发布与 caichip 服务端 **分离版本**；通过 **环境变量** 连接 `caichip_base_url`，与 `scripts/` 下爬虫脚本 **解耦**。

## 5. 端到端流程（串联）

1. Agent 启动 → **Python 环境自检**（缺失则 **自举安装**）→ **pip 自检**（缺失则 **自举安装 pip**，见 §4.1.1）→ 注册/鉴权 → 同时启动 **任务心跳（10s）** 与 **脚本安装心跳（10min）** 两条周期逻辑。  
2. **脚本安装心跳** 携带已同步脚本列表 → caichip 比对 → 返回待同步脚本或空。  
3. Agent 下载 ZIP、校验哈希与 `version.txt` → **创建 venv**；若存在 `requirements.txt` 则 **安装依赖**，否则视为无依赖 → 写入 `env_status=ready`（或 `failed`）→ 下次 **脚本安装心跳** 上报版本 + 准备状态（可选：主动多打一次脚本安装心跳）。  
4. **任务心跳** 周期到达 → caichip 将待执行任务放入响应 → Agent **校验 `version` 与本地包、`env_status`** → **定位入口 Python 并执行**（**执行期间仍按周期发任务心跳**，§4.2）→ 上报结果。  
5. caichip 更新任务状态；若 Agent **离线**则对未终态任务 **重派**（§4.6、§6.4），并触发业务后续流程（如入库、告警）。

## 6. 非功能需求（概要）

| 类别 | 要求 |
|------|------|
| **安全** | **TLS**；**鉴权采用 API Key**（见下）；脚本包签名/哈希；最小权限。 |
| **可观测** | 两类心跳时间、脚本版本与 `env_status`、任务 ID、执行耗时与错误码可查询。 |
| **性能** | 心跳与任务载荷大小可控；大批量脚本可考虑 CDN/对象存储。 |

**鉴权模型（已约定：API Key）**

- caichip 为 Agent **颁发 API Key**（可按 Agent、环境或租户维度区分，以服务端设计为准）。  
- Agent 将 Key 保存在 **本地配置文件或环境变量**，**不得** 打入脚本 ZIP 或提交到仓库。  
- 调用任务心跳、脚本安装心跳及任务结果上报等接口时，在 **HTTP Header** 中携带（例如 `Authorization: Bearer <api_key>` 或 `X-API-Key: <api_key>`，**以最终实现与 OpenAPI 为准**）。  
- 服务端应支持 **吊销、轮换**；泄露时作废旧 Key 并下发新 Key。

### 6.1 长轮询、HTTP 超时与 Agent 在线（已约定）

| 项 | 说明 |
|----|------|
| **长轮询** | 两类心跳均为 **REST + HTTP 长轮询**；OpenAPI 须固定：**服务端最长挂起时间**、无任务/无同步项时响应体。 |
| **HTTP 超时对齐** | Agent 侧 **HTTP Client 总超时** 须 **大于** 服务端单次挂起上限（避免未收到响应即断开）；详见 **[分布式采集Agent-API协议](./分布式采集Agent-API协议.md) §6**。 |
| **在线判定** | 以 **任务心跳** 为 **主在线信号**；**默认**在 **连续 120 秒**（2 分钟）内无 **成功** 任务心跳则判 **挂起/离线**（§4.2）；脚本安装心跳 **不参与** 该阈值。 |
| **与心跳间隔的关系（已约定）** | 若任务心跳周期配置为 **`interval_sec`**（默认 `10`），离线阈值 **`T_offline_sec`** 须随间隔放大，避免 **间隔过大** 时未等够周期即判离线：建议 **`T_offline_sec = max(120, 6 × interval_sec)`**（**6** 为可配置系数，OpenAPI/服务端配置为准）。**若 `interval_sec > 120`**，必须以 **`T_offline ≥ 6 × interval_sec`** 或等价规则重算，**不得** 仍用固定 120s 作为唯一判据。 |

### 6.2 任务执行超时与进程组终止（已约定）

| 项 | 说明 |
|----|------|
| **超时来源** | **`timeout_sec`** 由 **任务下发** 指定；**未下发或无效则默认 `300`（5 分钟）**。 |
| **执行与计时** | **仅 Agent** 对任务子进程计时；服务端不对执行中子进程计时。 |
| **终止方式** | 超时时 Agent **终止该任务对应的进程组**（含子进程），如 POSIX **进程组** / **Windows Job Object** 等（以实现为准）；**禁止** 仅杀主进程导致子进程残留。 |
| **结果上报** | **`status=timeout`**，附 stdout/stderr 摘要（截断见 **§6.6**）。 |

### 6.3 幂等与 `task_id`（已约定）

| 项 | 说明 |
|----|------|
| **重复上报** | 网络重试导致 **同一执行** 多次上报 → 服务端 **幂等**，不重复触发业务副作用。 |
| **重复执行** | 服务端再次下发 **同一 `task_id`** 时，Agent **允许再次执行** 并再次上报；服务端用 **`attempt`** 或时间序区分轮次，或 **仅保留最后一次结果**——以 OpenAPI 为准。 |

### 6.4 离线任务重派与「迟到结果」（已约定）

当 Agent 被判 **挂起/离线**（§4.2、§6.1）时，caichip 对 **尚无终态结果** 的任务进行 **重派**（重新入队或分配给其他 Agent）。

| 项 | 说明 |
|----|------|
| **重派后迟到结果** | 任务已 **重派** 且 **新执行已产生终态**（或任务已 **取消/关闭**）后，**旧 Agent** 仍就 **原派发轮次** `POST` 任务结果 → 服务端应 **拒收或标记无效**（如 **HTTP 409** `TASK_REASSIGNED` / `LEASE_EXPIRED`，见 API 协议 **§4.2**），**不得** 覆盖新执行结果。 |
| **租约/代数（建议）** | OpenAPI 可为每次派发增加 **`lease_id`** 或 **`dispatch_generation`**；结果上报 **须携带** 与当前执行一致的租约，否则服务端拒收。 |
| **幂等** | 与 **§6.3** 配合：**同一 `task_id` + 同一租约** 的重复上报仍 **幂等**；**不同租约** 表示不同执行轮次，由 **`attempt`** 等区分。 |

### 6.5 脚本包版本字符串（`version.txt` 与任务 `version`）（已约定）

| 项 | 说明 |
|----|------|
| **格式** | **`version.txt`** 为 **单行** 版本字符串；**允许** 可选前缀 **`v` 或 `V`**（如 `v1.2.0`）。 |
| **比对** | 与 caichip 目标版本、心跳中的 `version`、任务载荷中的 **`version`** 均应先 **规范化**（去首尾空白；若以 `v`/`V` 开头且表示版本号则 **去掉该前缀** 再比较），避免 `1.2.0` 与 `v1.2.0` 分叉。建议 **SemVer**，具体解析以实现为准。 |
| **缺 `requirements.txt`** | **视为无依赖**（跳过 `pip install -r`，不判校验失败；仍须完成 ZIP/`version.txt` 等校验与 venv 创建，见 §2、§4.4）。 |

### 6.6 结果与日志载荷大小（已约定）

| 项 | 说明 |
|----|------|
| **目的** | 防止 **单任务** 超大 JSON/日志 压垮网关、DB 与排障体验。 |
| **建议上限** | **`result`** 序列化后、**`stdout_tail`/`stderr_tail`** 合计等，由 OpenAPI/服务端配置 **单请求最大字节数**（如 **`result` ≤ 1MB**、**每路 tail ≤ 32KB**，可调整）；超限由 Agent **截断**并可选附 **`truncated: true`** 或哈希占位。 |
| **对齐** | 与 API 协议 **§4.1** 字段说明一致。 |

### 6.7 同一 `script_id` 多版本目录与磁盘清理（已约定）

| 项 | 说明 |
|----|------|
| **并存** | 本地以 **`script_id`/`version`/** 为目录可 **多版本并存**；任务 **仅使用** 载荷中 **`version`** 对应目录。 |
| **清理** | Agent 或运维策略应对 **不再被引用** 的旧版本做 **保留 N 个版本 / 最大磁盘占用 / 按年龄删除**（具体算法与默认以 OpenAPI 或 Agent 配置为准），与 **§8.4**「磁盘」项一致。 |

### 6.8 Agent 优雅退出与执行中任务（已约定）

| 项 | 说明 |
|----|------|
| **目标** | 进程收到 **SIGTERM / Windows 停止服务** 时，**尽量** 先 **停止接收新任务**、对运行中任务 **先发终态或显式撤销**（若协议支持），再 **终止进程组**（**SIGTERM → 宽限期 → SIGKILL**，与 **§6.2** 硬超时区分）。 |
| **最长等待** | **宽限期**（如 30s～与当前 `timeout_sec` 取小）由 OpenAPI 或 Agent 配置固定；超时仍 **强制杀进程组** 并退出。 |

## 8. 评审补充项（后续版本建议落稿）

以下条目为 **OpenAPI / 详细设计** 中仍待细化的内容；**已定稿** 的约定已上移至 **§4、§6**，本节 **不重复**。

### 8.1 协议与接口

| 项 | 说明 |
|----|------|
| **JSON 契约** | 两类心跳、任务结果上报、脚本下载相关接口的 **请求/响应字段**、错误体、`Content-Type`、URL 版本前缀（如 `/api/v1`）建议以 OpenAPI 固定。 |
| **任务结果上报** | 明确 **独立接口**（如 `POST .../task/result`）是否必选；与任务心跳「捎带」的关系；失败 **重试与 ACK** 策略。 |
| **脚本包下载** | 下载 URL 的 **鉴权**（临时签名 vs Header）、**过期时间**、是否支持 **断点续传**。 |

**说明**：**长轮询参数**、**HTTP 超时对齐**、**Agent 在线判定**、**幂等与 `task_id`**、**重派与租约**、**结果大小**、**多版本清理**、**优雅退出** 见 **§6.1、§6.3–§6.8**；与 API 协议 **§4、§6** 一致。

### 8.2 Agent 运行时

| 项 | 说明 |
|----|------|
| **注册** | `agent_id` 与 API Key 是 **预置** 还是 **首次注册领取**；绑定关系。`agent_id` **默认由本机 MAC 推导**（见 §4.1「身份」），与 API Key 的绑定方式待 OpenAPI 固定。 |
| **参数注入** | 任务 `参数 JSON` 传给子进程的 **方式**（环境变量 / 命令行 / stdin）及 **大小上限**。 |
| **优雅退出** | 正文 **§6.8** 已约定宽限期与进程组终止；OpenAPI 可再固定信号与默认值。 |
| **Agent 自升级** | Agent **二进制** 的升级/回滚路径（与脚本 ZIP 下发区分）。 |

**说明**：**任务并发（同类 = `script_id`）**、**执行中任务心跳** 见 **§4.2、§4.5**；全局并行上限与异步策略待 OpenAPI。

### 8.3 安全

| 项 | 说明 |
|----|------|
| **`entry_file`** | 仅允许包内 **规范化相对路径**，**禁止 `..`** 跳出脚本根目录。 |
| **TLS** | 是否 **强制校验证书**、内网 HTTP 例外是否允许。 |
| **密钥存储** | 配置文件权限；是否对接 **主机密钥管理**。 |

### 8.4 可观测与运维

| 项 | 说明 |
|----|------|
| **指标 / 日志** | 关键事件（心跳失败、下载失败、pip 失败、任务起止）；可选 **Prometheus 指标** 命名约定。 |
| **追踪** | `trace_id` / `request_id` 是否贯穿任务全链路。 |
| **磁盘** | **§6.7** 已约定多版本清理方向；OpenAPI 可固定默认 **保留 N 版 / 最大占用**。 |

### 8.5 脚本包与任务语义

**已定稿内容** 已上移至 **§2、§4.4、§4.5、§6.2、§6.5**（`version.txt`、任务 `version`、缺 `requirements.txt`、超时与进程组终止等），本节仅保留占位，避免与正文重复。

### 8.6 与现有 caichip 集成

- **正文已约定**：与现有 caichip 仓库、YAML/演进配置源的集成方式见 **§4.7**；本节其余 OpenAPI 级细节（具体路径名、环境变量名表）仍在详细设计中补齐。
